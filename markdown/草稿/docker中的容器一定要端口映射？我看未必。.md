# 当Windows本地Redis与Docker的Redis容器共享6379端口：一场诡异的和平共处

## 引言

在开发过程中，我们经常会遇到需要在同一台机器上运行多个Redis实例的场景。最近，我在Windows 11本机上安装了一个Redis服务（监听6379端口），同时使用Docker Desktop启动了一个Redis容器，并尝试将容器的6379端口也映射到主机的6379端口。按照常规理解，端口冲突应该会导致Docker容器启动失败，然而事实却让人意外：`docker-compose up`顺利执行，没有任何报错。

但当我尝试连接时，发现`redis-cli`和RESP.app都连接到了本机的Redis，而不是Docker容器。直到我将Docker的端口映射改为`6380:6379`，才能正常区分和管理两个Redis实例。这背后的原因是什么？本文将结合网络端口绑定机制和Docker的网络原理，深入分析这一现象。

## 环境说明

- 操作系统：Windows 11
- 本地Redis：Windows原生运行的`redis-server.exe`，监听6379端口
- Docker Desktop：基于WSL2后端
- Docker镜像：`redis:7-alpine`
- 编排工具：docker-compose

关键配置文件（`docker-compose.yml`）：
```yaml
redis:
  image: redis:7-alpine
  container_name: docker-redis
  restart: always
  ports:
    - "6379:6379"
```

## 现象重现

执行`docker-compose up -d`后，容器正常启动，且`docker ps`显示端口映射为`0.0.0.0:6379->6379/tcp`。

查看Windows端口占用：
```cmd
netstat -ano | findstr :6379
```

输出：
```
TCP    0.0.0.0:6379           0.0.0.0:0              LISTENING       5168
TCP    127.0.0.1:6379         0.0.0.0:0              LISTENING       6388
TCP    127.0.0.1:6379         127.0.0.1:31379        ESTABLISHED     6388
TCP    127.0.0.1:31379        127.0.0.1:6379         ESTABLISHED     42500
TCP    [::]:6379              [::]:0                 LISTENING       5168
TCP    [::1]:6379             [::]:0                 LISTENING       26412
```

进一步查看进程：
```cmd
tasklist | findstr 6388   # 本地redis-server.exe
tasklist | findstr 5168   # com.docker.backend.exe
```

结果：
- PID 6388 → `redis-server.exe`（本地Redis）
- PID 5168 → `com.docker.backend.exe`（Docker Desktop后端）

尝试连接：
```bash
redis-cli -h 127.0.0.1 -p 6379 INFO server
```
返回的是win机器本地Redis的版本信息。使用RESP.app连接同样指向本地Redis。

## 原因分析：IP地址绑定与端口复用规则

### 1. 端口冲突的本质是“地址+端口”的组合

很多人误以为一个端口只能被一个进程占用，但实际上，**TCP/UDP端口是通过“IP地址 + 端口号”来唯一标识一个监听端点的**。也就是说，同一个端口号可以绑定到不同的IP地址上，被不同的进程分别监听。

在Windows（以及其他主流操作系统）中，常见的IP绑定方式有：

- `0.0.0.0`：通配地址，表示监听本机所有可用的IPv4接口（包括127.0.0.1、局域网IP、公网IP等）。
- `127.0.0.1`：仅监听本地回环接口。
- 具体的局域网IP（如192.168.1.100）：仅监听该网卡地址。

### 2. netstat输出解读

从上面的输出可以看到：

- **PID 5168（Docker后端）** 监听在 `0.0.0.0:6379` 和 `[::]:6379`（IPv6通配）。这意味着Docker希望将到达主机**任意IPv4地址**（包括127.0.0.1）的6379端口的流量，都转发给容器。
- **PID 6388（本地Redis）** 监听在 `127.0.0.1:6379`，**仅**接收发往本地回环地址的请求。

这两者并不冲突，因为它们绑定的IP地址不同。一个是通配地址（覆盖所有），另一个是具体的回环地址。在Windows中，允许这样的“重叠”绑定，只要具体的绑定先于通配绑定生效（具体规则取决于协议栈实现，通常更具体的绑定具有更高优先级）。

### 3. 为什么连接127.0.0.1:6379会优先到达本地Redis？

当客户端使用`127.0.0.1:6379`发起连接时，操作系统内核在查找匹配的监听套接字时，会遵循**“最具体匹配原则”**：

- 存在一个明确绑定到`127.0.0.1:6379`的套接字（本地Redis）
- 也存在一个绑定到`0.0.0.0:6379`的套接字（Docker转发器）

由于`127.0.0.1`比`0.0.0.0`更具体（通配符相当于任何地址，但具体的回环地址优先级更高），内核会将连接交给本地Redis处理。因此，无论Docker容器是否在运行，只要本地Redis占据了`127.0.0.1:6379`，所有针对该地址+端口的流量都会被本地Redis拦截。

### 4. Docker端口映射的底层实现

在Windows上的Docker Desktop（使用WSL2或Hyper-V后端）中，端口映射并不是直接让容器内的进程绑定到主机的网络堆栈。而是通过一个用户态的后端进程（`com.docker.backend.exe`或`vpnkit`）在主机上创建一个代理监听器（绑定到`0.0.0.0:6379`），将收到的TCP流量通过虚拟网络转发到容器的6379端口。

因此，`0.0.0.0:6379`上的是Docker的转发代理，而非容器内的Redis进程本身。这个代理与本地Redis的`127.0.0.1:6379`可以共存。

## 解决方案

如果需要同时运行两个Redis实例并分别管理，有以下几种方式：

### 方案一：修改其中一个Redis的映射端口（推荐）

将Docker的端口映射改为其他未占用的端口，例如：
```yaml
ports:
  - "6380:6379"
```
这样，本机Redis依然在6379，Docker Redis在6380，互不干扰。使用`redis-cli -p 6379`连接本地，`redis-cli -p 6380`连接容器。

### 方案二：使用不同的绑定地址

让本地Redis只绑定到某个非回环地址（如局域网IP），或者让Docker只绑定到特定地址。例如：
```yaml
ports:
  - "192.168.1.100:6379:6379"
```
但这会增加配置复杂度，且不便于本地回环测试。

### 方案三：停止本地Redis服务，释放127.0.0.1:6379

如果不再需要本机Redis，可以直接停止其服务。之后Docker的`0.0.0.0:6379`监听器将正常接收发往127.0.0.1的请求，连接到容器内的Redis。

## 总结

| 关键点 | 说明 |
|--------|------|
| 端口冲突规则 | 冲突基于IP地址+端口组合，不同IP地址可绑定同一端口 |
| Docker映射行为 | 在主机上创建`0.0.0.0:6379`的代理转发器 |
| 本地Redis行为 | 通常默认绑定`127.0.0.1:6379` |
| 连接优先级 | 客户端连接`127.0.0.1:6379`时，更具体的`127.0.0.1`绑定优先于通配`0.0.0.0` |
| 解决方案 | 修改映射端口，或调整绑定地址，或停止冲突服务 |

这次排查经历提醒我们，端口冲突并不总是表现为“Address already in use”的错误。在某些情况下，多个进程可以“和平共处”，但实际路由结果可能不符合预期。理解网络协议栈的绑定和匹配规则，能帮助我们快速定位这类隐蔽的问题。

希望这篇文章对你在Windows + Docker环境下的开发有所帮助。如果你也遇到过类似“端口不冲突但访问不到预期服务”的怪现象，不妨从绑定地址的角度去审视一下。
