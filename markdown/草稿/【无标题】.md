# 开源项目二开，为什么我更推荐 `git clone --depth 1`

当你执行一个 `git clone` 时，你真的需要把这个项目过去 5 年甚至 10 年的所有提交历史都拉下来吗？

大多数情况下——**不需要**。

尤其是在二次开发（fork / 魔改 / 本地跑项目）场景下，这些历史不仅没有帮助，反而拖慢你的开发节奏。

这也是为什么我在做开源项目二开时，默认会使用：

```bash
git clone --depth 1 <repo>
```

这篇文章就讲清楚一件事：**为什么这是一个更高效的默认选择。**

---

## 什么是 `git clone --depth 1`？

```bash
git clone --depth 1 https://github.com/mealie-recipes/mealie.git
```

这个命令的含义是：

> 只克隆仓库**最近一次提交**，不包含历史记录。

更准确地说：

* 默认 `git clone`

  * 拉取完整提交历史
  * 获取所有分支和 tag 的引用
  * 只 checkout 默认分支（如 `main`）

* 使用 `--depth 1`

  * 只拉取默认分支的**最新快照**
  * `.git` 目录是一个 **shallow repository（浅仓库）**

👉 重要区别：

> 工作区代码是一样的，但仓库“历史是不完整的”。

---

## 二次开发场景下的三大优势

### 1. 极速下载，节省带宽

很多大型项目：

* Linux 内核
* Kubernetes
* Android

`.git` 目录可能达到：

* 几百 MB
* 甚至几个 GB

而 `--depth 1`：

* 只下载当前状态
* 通常可以减少 **90%+ 数据量**

👉 从“几分钟” → “十几秒”，非常常见。

---

### 2. 节省磁盘空间

完整 clone：

* 包含所有历史对象
* 多个项目时占用巨大

浅克隆：

* 只保留当前版本
* 对于多项目开发环境非常友好

例如：

* 完整 clone：200MB+
* `--depth 1`：可能 < 20MB

---

### 3. 非常适合 CI/CD 和临时环境

在这些场景中：

* GitHub Actions
* GitLab CI
* Docker 构建

你通常只是为了：

* build
* test
* package

👉 根本不需要历史记录。

实际上：

> GitHub Actions 的 `actions/checkout` 默认就是浅克隆（`fetch-depth: 1`）

---

## 对二次开发的影响：真的够用吗？

答案是：**绝大多数情况完全够用。**

你仍然可以：

* ✅ 修改代码并运行
* ✅ `git add` / `git commit`
* ✅ 创建新分支
* ✅ `git push`（大多数平台没问题）
* ✅ 发起 Pull Request

👉 需要注意的是：

> 在部分开启严格校验的仓库中，push 可能要求完整历史（可以后续补全）

---

### 唯一的限制

浅克隆的核心限制是：

* ❌ `git log` 几乎没有历史
* ❌ `git blame` 无法追溯
* ❌ 无法直接切换到旧版本

但对于典型二开任务：

* 修 bug
* 改配置
* 加功能

👉 **完全不影响**

---

## 常见问题 & 解决方案

| 问题          | 解决方案                                  |
| ----------- | ------------------------------------- |
| 想查看历史提交     | `git fetch --unshallow`               |
| 想补充部分历史     | `git fetch --depth=100`               |
| push 被拒（少见） | 先 `unshallow` 再 push                  |
| 想切换其他分支     | `git fetch origin <branch> --depth 1` |

👉 核心思想很简单：

> **需要历史 → 再补，而不是一开始就全部拉下来**

---

## 进阶：比 `--depth 1` 更灵活的方式

`--depth 1` 只是最激进的浅克隆方式，你还有更多选择：

### 1. 保留部分历史

```bash
git clone --depth 50 <repo>
```

适用于：

* 想看最近变更
* 但不想要完整历史

---

### 2. 只克隆单分支

```bash
git clone --depth 1 --single-branch <repo>
```

进一步减少数据量（适合大仓库）

---

### 3. 部分克隆（进阶）

```bash
git clone --filter=blob:none <repo>
```

特点：

* 不下载文件内容（按需拉取）
* 适用于超大 monorepo

👉 这是更高级的优化手段（多数人用不到，但值得知道）

---

## 什么时候不该用 `--depth 1`？

以下场景建议使用完整 clone 或增加 depth：

* 需要 `git blame` 分析代码来源
* 需要在多个历史版本之间切换
* 做代码审计 / 演进分析
* 深度参与开源贡献

👉 折中方案：

```bash
git clone --depth 100 <repo>
```

---

## 总结

| 场景             | 推荐方式             |
| -------------- | ---------------- |
| 快速二开 / 本地运行    | `--depth 1`      |
| CI/CD / Docker | `--depth 1`      |
| 需要部分历史         | `--depth 50~100` |
| 深度参与项目         | 完整 clone         |

---

## 最后一句

二开开源项目的本质是：

> **站在“当前代码”上解决问题，而不是背着整个历史包袱。**

所以更高效的默认方式应该是：

```bash
git clone --depth 1 <repo>
```

如果哪天你真的需要历史，再执行：

```bash
git fetch --unshallow
```

👉 **先轻装上阵，再按需补全。**
